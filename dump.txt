Checkout  1. git push Purpose: Uploads (pushes) your local commits to a remote repository (like GitHub, GitLab, etc.) so others can access your latest work. Syntax: git push <remote_name> <branch_name> remote_name → usually origin (the default name of the remote repo) branch_name → usually main or master, but it could be any branch Example: git push origin main 2. git merge Purpose: Combines changes from one branch into another — commonly used when you finish a feature branch and want to merge it into the main branch. Syntax: git merge <branch_name>  3. git commit Purpose: Records the staged changes (those added using git add) into the local repository with a descriptive message. Syntax: git commit -m "commit message" -m → specifies the commit message inline The message should briefly describe what you changed 4. git checkout Purpose: Used to switch between branches or restore files to a previous state. Syntax (to switch branch): git checkout <branch_name> Syntax (to create and switch to a new branch): git checkout -b <new_branch_name> | 8 | L2 | CO2 PO1

The first developer commits the code on the local repository and pushes it to the remote repository The local repository is the developer’s working copy on their own computer. The developer makes changes (adds new files or modifies existing ones). 2. The second developer gets the pushed code from the remote repository The remote repository now contains the latest version of the project. The second developer needs to make sure they have the same version before starting any new work.    3. The second developer updates this code, creates a commit, and pushes the new version of the code to the remote repository Now the second developer works on the latest version of the code — perhaps fixing a bug, improving performance, or adding a new feature. 4. Finally, the first developer retrieves the latest version of the code in the local repository The first developer’s local copy is now out of date since the second developer has pushed new changes. 2 | b) | Consider the scenario which involves setting up a mechanism that notify Jenkins as soon as a new push occurs in the git hub repository. Analyze the above using GitHub webhook. In order for Jenkins to run a new job, we must first create a webhook in the GitHub repository. This webhook will be used to notify Jenkins as soon as a new push occurs in the repository. To do this, follow these steps:  1. In the GitHub repository, go to the Settings | Webhooks menu. 2. Click on the Add Webhook button. 3. In the Payload URL field, fill in the URL address of Jenkins followed by /githubwebhook/, leave the secret input as it is, and choose the Just the push event option.   Payload URL is the endpoint (web address) where GitHub sends a POST request whenever an event (like a code push) happens. This payload contains a JSON message with all the details about the event — who made the change, which branch was updated, commit details, etc. Jenkins receives this payload and decides whether to trigger a build job based on the configuration. he general format is: http://<JENKINS_SERVER_URL>:<PORT>/github-webhook/ or if Jenkins uses HTTPS: https://<JENKINS_SERVER_URL>/github-webhook/  4. Validate the webhook. | 7 | L4 | CO3 PO2
